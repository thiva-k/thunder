# OAuth 2.0 Configuration

This reference covers all OAuth 2.0 configuration options available for registered clients in <ProductName />, including grant types, client authentication methods, token settings, and advanced security options.

## Grant Types

The `grantTypes` field lists the OAuth 2.0 flows the client may use. Select only the flows your application requires.

| Grant Type | Description | Typical Use |
|---|---|---|
| `authorization_code` | Authorization Code flow. The client receives an authorization code and exchanges it for tokens at the token endpoint. | User-facing applications: browser apps, mobile apps, full-stack apps. |
| `refresh_token` | The client exchanges a refresh token for a new access token without re-authenticating the user. Use alongside `authorization_code`. | Applications that require long-lived sessions. |
| `client_credentials` | Machine-to-machine flow. The client authenticates with its own credentials â€” no user interaction. | Backend services, daemons, and internal APIs. |
| `urn:ietf:params:oauth:grant-type:token-exchange` | Token Exchange (RFC 8693). The client exchanges one token for another, optionally narrowing scopes or acting on behalf of a user in a service-to-service call. | Service mesh scenarios and delegated authorization. |

## Response Types

The `responseTypes` field lists the response types the client may request in an authorization request.

| Response Type | Description |
|---|---|
| `code` | Returns an authorization code. Exchange the code for tokens at the token endpoint. |

## Token Endpoint Authentication

The `tokenEndpointAuthMethod` field controls how the client proves its identity at the `/oauth2/token` endpoint.

| Method | Description | Use When |
|---|---|---|
| `client_secret_basic` | Sends credentials as a Base64-encoded `Authorization: Basic` header. Default for confidential clients. | Server-side applications that store a secret securely. |
| `client_secret_post` | Sends `client_id` and `client_secret` in the request body. | Clients that cannot set custom HTTP headers. |
| `private_key_jwt` | The client signs a JWT assertion with its private key. <ProductName /> verifies the signature using the client's registered certificate. Requires a `JWKS` or `JWKS_URI` certificate on the application. | High-security deployments where secret-based authentication is not acceptable. |
| `none` | No client authentication. Used for public clients that cannot store a secret securely. | Browser apps and mobile apps. |

## PKCE

`pkceRequired` controls whether the client must include a `code_challenge` in every authorization request, as defined in RFC 7636.

<ProductName /> automatically sets `pkceRequired` to `true` for public clients (`publicClient: true`). For confidential clients, enabling PKCE adds protection against authorization code interception attacks.

## Scopes

The `scopes` field lists the OAuth 2.0 scopes this client may request. <ProductName /> rejects requests for scopes outside this list.

## Scope-to-Claims Mapping

The `scopeClaims` field maps scope names to the user attributes returned in tokens and userinfo responses when that scope is requested.

```json
{
  "scopeClaims": {
    "profile": ["given_name", "family_name", "picture"],
    "email": ["email", "email_verified"]
  }
}
```

When a client requests the `profile` scope, <ProductName /> includes `given_name`, `family_name`, and `picture` in the ID token and userinfo response. Scopes without an entry in `scopeClaims` return no extra claims.

## Token Configuration

The `token` field controls the validity periods and user attributes embedded in access tokens and ID tokens.

### Access Token

| Field | Type | Description |
|---|---|---|
| `token.accessToken.validityPeriod` | integer | Access token lifetime in seconds. Default: `3600`. |
| `token.accessToken.userAttributes` | array | User attributes to embed in the access token payload. Standard claims (`sub`, `iss`, `exp`, and others) are always included and cannot be removed. |

### ID Token

| Field | Type | Description |
|---|---|---|
| `token.idToken.validityPeriod` | integer | ID token lifetime in seconds. Default: `3600`. |
| `token.idToken.userAttributes` | array | User attributes to embed in the ID token. Only attributes covered by the requested scopes are returned. |
| `token.idToken.responseType` | string | Response format: `JWT` (signed only, default), `JWE` (encrypted), or `NESTED_JWT` (signed then encrypted). |
| `token.idToken.encryptionAlg` | string | Key-management algorithm. Required for `JWE` and `NESTED_JWT`. Supported values: `RSA-OAEP`, `RSA-OAEP-256`. |
| `token.idToken.encryptionEnc` | string | Content-encryption algorithm. Required when `encryptionAlg` is set. Supported values: `A128CBC-HS256`, `A256GCM`. |

ID token encryption requires a certificate on the application. See [Certificate Configuration](#certificate-configuration).

## Userinfo

The `userInfo` field controls what the `/userinfo` endpoint returns and in what format.

| Field | Type | Description |
|---|---|---|
| `userInfo.userAttributes` | array | Attributes returned from `/userinfo`. Configure independently or mirror the ID token selection. |
| `userInfo.responseType` | string | Response format: `JSON` (default), `JWS` (signed), `JWE` (encrypted), or `NESTED_JWT` (signed then encrypted). |
| `userInfo.signingAlg` | string | Signing algorithm. Required for `JWS` and `NESTED_JWT`. Supported values: `RS256`, `RS512`, `PS256`, `ES256`, `ES384`, `ES512`, `EdDSA`. |
| `userInfo.encryptionAlg` | string | Key-management algorithm. Required for `JWE` and `NESTED_JWT`. Supported values: `RSA-OAEP`, `RSA-OAEP-256`. |
| `userInfo.encryptionEnc` | string | Content-encryption algorithm. Required when `encryptionAlg` is set. Supported values: `A128CBC-HS256`, `A256GCM`. |

Signing and encryption require a certificate on the application.

## Certificate Configuration

The `certificate` field attaches a public key to the application. A certificate is required for:

- `private_key_jwt` token endpoint authentication
- ID token encryption
- Userinfo signing or encryption

| Certificate Type | Description |
|---|---|
| `None` | No certificate configured. |
| `JWKS` | Provide the JSON Web Key Set inline as a JSON object. |
| `JWKS_URI` | Provide the URL of the application's JWKS endpoint. <ProductName /> fetches the public keys from the URL to verify signed requests. The URL must use HTTPS. |

You cannot set both `JWKS` and `JWKS_URI` on the same application. Providing both returns an error.

## Pushed Authorization Requests

`requirePushedAuthorizationRequests` controls whether the client must submit authorization parameters to the `/oauth2/par` endpoint before starting the authorization flow, as defined in RFC 9126.

When set to `true`, <ProductName /> rejects any authorization request that arrives directly at the authorization endpoint without a prior PAR submission. This prevents sensitive request parameters from appearing in browser URLs or server logs.

You can also enforce Pushed Authorization Requests (PAR) globally for all clients by setting `oauth.par.require_par: true` in `deployment.yaml`. The per-application flag adds an additional per-client enforcement layer on top of the global setting.

## Default ACR Values

The `acrValues` field sets the default Authentication Context Class Reference (ACR) values for authorization requests that do not include `acr_values`. <ProductName /> applies these values when selecting which authentication flow to run.

```json
{
  "acrValues": ["urn:mace:incommon:iap:silver"]
}
```
